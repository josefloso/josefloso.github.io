State Isolation: Workspaces vs File Layouts — When to Use Each

Workspace Setup

Output of:

terraform workspace list

* dev
  staging
  prod

S3 State File Paths per Workspace

Assuming backend config:

key = "env:/terraform.tfstate"

Actual paths in S3:


env:/dev/terraform.tfstate
env:/staging/terraform.tfstate
env:/prod/terraform.tfstate

Each workspace automatically prefixes the state path with its name.

Workspace Configuration Code


resource "aws_instance" "example" {
  ami           = "ami-12345678"

  instance_type = terraform.workspace == "prod" ? "t3.large" : "t3.micro"

  tags = {
    Name = "${terraform.workspace}-instance"
  }
}

This makes:

  • dev → t3.micro
  • staging → t3.micro
  • prod → t3.large

Backend Configs

dev/backend.tf


terraform {
  backend "s3" {
    bucket         = "tf-state-bucket"
    key            = "dev/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-locks"
  }
}

prod/backend.tf


terraform {
  backend "s3" {
    bucket         = "tf-state-bucket"
    key            = "prod/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-locks"
  }
}

Why key paths matter:

  • Each environment has a separate state file
  • No accidental overlap
  • No switching environments without changing directories

Remote State Data Source

Configuration


data "terraform_remote_state" "network" {
  backend = "s3"

  config = {
    bucket = "tf-state-bucket"
    key    = "dev/network/terraform.tfstate"
    region = "us-east-1"
  }
}

Referencing an output


resource "aws_instance" "example" {
  subnet_id = data.terraform_remote_state.network.outputs.subnet_id
}

Workspace vs File Layout Comparison

Which provides stronger environment isolation?

File layout.

  • Separate directories
  • Separate backend configs
  • No shared execution context

Which is easier to accidentally misconfigure?

Workspaces.

  • You can be in the wrong workspace
  • Same code, different outcomes

Which scales better across a large team?

File layout.

  • Clear boundaries per environment
  • Easier ownership
  • CI/CD friendly

Which would I use in production?

File layout.

State Locking in Multi-Environment Context

DynamoDB locking behavior:

  • Lock is tied to the state file key

With workspaces:


env:/dev/terraform.tfstate → separate lock
env:/prod/terraform.tfstate → separate lock

Conclusion:

  • No cross-workspace locking
  • Locking is scoped to state path

Chapter 3 Learnings

What is terraform_remote_state?

It allows one Terraform configuration to read outputs from another configuration’s state file.

Limitations

  • Tight coupling between configurations
  • No fine-grained access control
  • Breaks if state location changes
  • Not suitable for sensitive data sharing

Challenges and Fixes

Issue 1: Wrong workspace applied

Fix:

terraform workspace show

Issue 2: Backend reinitialization errors

Fix:

terraform init -reconfigure

Issue 3: Confusing state paths

Fix:

env:/<workspace>/terraform.tfstate

Issue 4: Directory duplication

Fix:

  • Use shared modules
  • Accept backend duplication for isolation

Final Verdict

Workspaces:

  • Fast
  • Simple
  • Flexible
  • Risk of mistakes

File layouts:

  • Safer
  • More structured
  • Better for teams

Recommendation: Use file layouts for production. Use workspaces for experiments or small setups.
